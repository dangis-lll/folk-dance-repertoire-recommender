# 中国民族民间舞剧目智能查询系统

这是一个先用 Python 实现的 MVP：Flask + SQLite + DeepSeek API。系统不存视频本体，只存剧目信息、视频链接、专家评分、推荐记录。

## 功能

- 剧目库：新增、编辑、查询剧目
- 视频版本：同一个剧目可挂多个平台链接
- 专家评分：按身体要求、学习成本、表演要求、使用场景打分
- 帮我选剧目：自然语言输入学生条件，系统从已确认的剧目和专家评分中筛选推荐
- DeepSeek 接入：读取环境变量 `DEEPSEEK_API_KEY`
- 批量导入：内部建库时批量导入剧目和视频链接
- 数据采编工作台：内部模式下 B站关键词自动搜索，结果临时展示，人工确认后入库
- 机构模式：隐藏自动搜索，只保留手动添加视频链接
- 视频浏览：YouTube/B站优先内嵌预览，其他平台保留跳转
- 推荐记录：保存每次推荐的输入、推荐数量和推荐快照

## 身体条件与推荐

专家评分里的身体要求会进入推荐计算，包括柔韧、力量、控制、耐力、协调性、脚下技术、旋转跳跃和入门门槛。用户输入“软度中等、协调性弱、耐力好、技巧偏弱”等描述时，系统会解析成对应能力层级，并参与“能力匹配”分数。

评分表里的“是否需要专业教师指导”使用“是/否”选择；数据库内部仍保存为 `1/0`，其中 `1` 表示建议专业教师把控，`0` 表示普通教学也可处理。

## 启动

本机优先使用 `pt` conda 环境：

```powershell
conda run -n pt python app.py
```

打开：

```text
http://127.0.0.1:5000
```

第一次启动会自动创建 `dance_repertoire.db` 并写入示例数据。

## DeepSeek 配置

PowerShell 示例：

```powershell
$env:DEEPSEEK_API_KEY="你的 deepseek key"
conda run -n pt python app.py
```

可选模型变量：

```powershell
$env:DEEPSEEK_MODEL="deepseek-chat"
```

如果没有配置 `DEEPSEEK_API_KEY`，系统会自动使用本地规则解析，推荐功能仍可运行。

## 搜视频入库

数据采编/手动添加页在：

```text
http://127.0.0.1:5000/discover
```

默认是内部数据建设模式：

```powershell
$env:APP_MODE="internal"
```

内部模式下，B站自动搜索不需要额外配置。输入关键词后，系统会尝试从 B站搜索接口获取视频标题、链接、上传者、发布时间和缩略图。搜索结果只在页面临时展示，不会自动写入数据库。只有点击“入库”后，才会写入正式剧目和视频版本表。

机构交付模式：

```powershell
$env:APP_MODE="institution"
```

机构模式会隐藏 B站自动搜索，只保留手动添加视频链接、剧目库、专家评分、视频浏览和“帮我选剧目”。

YouTube、抖音、小红书不做自动 API 搜索。页面会生成这些平台的搜索入口，你可以打开平台搜索后，把合适的视频链接手动粘贴并确认入库。

入库时会自动做视频链接去重：

- B站按 BV 号规范化。
- YouTube 按 video id 规范化。
- 同一个平台视频 ID 已存在时，不会重复入库。

系统会自动整理剧目名称和别名，用来提示可能重复的剧目。

## 推荐算法

推荐不由 LLM 直接决定。流程是：

```text
用户自然语言
  -> DeepSeek/规则解析条件
  -> 结构化条件 + 全文索引召回候选
  -> 硬约束过滤
  -> 五维适配评分 + 少量老师反馈修正
  -> LLM 生成解释
```

五维评分：

```text
能力适配 30%
时间适配 20%
场景适配 25%
艺术价值 20%
视频参考质量 5%
风险惩罚：音乐/服装/教师依赖等
```

数据库会维护 `work_search_fts` 全文检索索引，索引内容来自剧目属性、专家摘要和视频标题/备注。推荐时先召回最多一批候选，再进入专家评分排序，避免数据量变大后每次遍历整库。

推荐页可以对单条结果点“合适 / 不合适 / 太难 / 太常见 / 视频资料不足”。这些反馈写入 `recommendation_feedback`，后续只作为轻微排序信号，不会替代专家评分。

推荐日志会保存完整推荐快照 `recommended_items_json`，方便复盘当时为什么推荐。

## 视频浏览

正式视频版本支持“浏览”：

- YouTube 链接会尝试使用 `https://www.youtube.com/embed/{video_id}` 内嵌播放。
- B站 BV 链接会尝试使用 `https://player.bilibili.com/player.html` 内嵌播放。
- B站搜索结果如果同时返回 `av` 链接和 `bvid`，系统会优先转成 `https://www.bilibili.com/video/BV...`，这样才能稳定生成内嵌播放器。
- 抖音、小红书等平台内嵌稳定性较差，系统会保留“在原平台打开”。

B站当前内嵌参数使用：

```text
isOutside=true
bvid=BV...
p=1
autoplay=0
danmaku=0
poster=1
```

只要链接里能解析出 BV 号，通常不需要额外获取 `cid`。

## 批量导入字段

支持这些字段：

```text
title, aliases, ethnicity, region, danceType, form, genderTendency,
platform, url, videoTitle, uploader, performer, thumbnailUrl, notes
```

最小示例：

```csv
title,aliases,ethnicity,region,danceType,form,genderTendency,platform,url,videoTitle,notes
心迹,"藏族舞心迹,女子独舞心迹",藏族,西藏,民族民间舞,独舞,女,B站,https://example.com/video,藏族舞《心迹》参考版本,替换为真实链接
```

## 推荐逻辑

推荐流程：

```text
自然语言需求
  -> DeepSeek JSON 解析，失败则本地规则解析
  -> SQLite 结构化筛选 + FTS 全文召回
  -> 基于专家评分排序
  -> DeepSeek 生成简短建议，失败则只显示规则推荐
```

LLM 不直接决定推荐结果，推荐结果仍来自数据库评分。

为了控制 token，系统不会把整库发给 LLM。LLM 只接收用户原始需求、解析后的条件，以及排序后的前几条候选摘要。搜索索引、候选召回、硬过滤和评分都在本地 SQLite/Python 完成。

## 数据治理

资料检查页：

```text
http://127.0.0.1:5000/quality
```

看板会集中显示：

- 疑似重复剧目
- 没有视频版本的剧目
- 没有专家评分的剧目
- 民族、形式或描述缺失的剧目
- 专家评分分歧较大的剧目
- 视频链接可能识别不完整的记录

新增剧目保存时会先做一次快速重复检测：

- 系统会把剧目名和别名标准化，去掉《》、空格、民族舞/独舞/剧目等常见描述词。
- 用标准化后的名称查询 `dance_work.normalized_title` 和 `dance_work_alias.normalized_alias`。
- 这两个字段都有索引，数据量变大后也能快速检测。
- 如果疑似重复，会先显示已有剧目；用户确认“仍然新建”后才会写入数据库。

剧目详情页提供“合并剧目”入口。合并时：

- 被合并剧目的视频版本迁移到主剧目
- 被合并剧目的专家评分迁移到主剧目
- 被合并剧目的别名迁移到主剧目
- 被合并剧目会从数据库删除，只保留主剧目
- 页面会在提交前弹出确认框，避免误操作

## 删除剧目

剧目详情页提供“删除剧目”。删除会直接从数据库移除：

- 剧目基础信息会删除。
- 该剧目下的视频版本会删除。
- 该剧目下的专家评分会删除。
- 页面会在提交前弹出确认框，避免误操作。
